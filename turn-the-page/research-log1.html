Preface

Before the afternoon.

Turn The Page is an app I've been building to solve a small but stubborn problem: musicians play with both hands, and sheet music still requires a third one. Existing page-turning apps rely on metronomes and timing guesses. I wanted one that actually listens — hears what the musician is playing, matches it against the score, and turns the page just before the last note of the page lands.

I'm not a traditional developer. I build with AI assistance and product intuition. The project had been moving: a command-line prototype that parsed MusicXML with music21, a Streamlit web app for testing, pitch detection running through aubio. Enough working pieces to feel like progress.

And yet, the core was failing quietly. Out of eight bars played on piano, the aligner would match only four to eight individual notes. I'd invented a forward-only search window to patch the cursor jumping backwards. I'd been tweaking tolerances and confidence thresholds for weeks. The numbers got better in tiny increments and then stopped improving.

This log is the afternoon I stopped tweaking and started questioning the architecture.

I · The critique

The roadmap was right about the ceiling.

I received a technical critique of my project roadmap. Reading it carefully turned out to be more valuable than a week of work, because it correctly named two problems I'd been circling without seeing:

1. Polyphony

aubio is a monophonic pitch detector. It expects one note at a time — flute, voice, monophonic violin line. My test pieces (Idea No. 10 by Gibran Alcocer, Una Mattina by Einaudi) are solo piano: chords, sustain pedal, overlapping voices. I had been asking a tool built for one sound to transcribe many. The ceiling wasn't in the tuning parameters. It was in the tool.

2. No real alignment

My "forward-only ±15 note search window" was a hand-rolled approximation of OLTW — Online Dynamic Time Warping — the standard algorithm in music information retrieval for real-time score following. I'd invented a clumsy cousin of the thing that already existed and had been studied for twenty years.

The problem wasn't that I needed better tuning. The problem was that I was building my own version of something a research community had already solved.

II · The discovery

A library published last autumn, built for exactly this.

Before acting on the critique, I verified its biggest claim: the existence of Matchmaker, supposedly an open-source Python library for real-time piano score following. If it was real and worked, it would collapse several phases of my roadmap at once.

It was real. Published at ISMIR 2025 by a team from KAIST and the Austrian Research Institute for AI. Apache 2.0 licensed (important, since aubio is GPL — a problem if I ever want to ship this as a commercial app). Designed to be "easily integrated into existing projects." The list of intended applications in the paper included, verbatim, "automatic page turning."

Key finding from the paper

The authors benchmarked several feature types. They found that an onset-sensitive spectral feature (LSE, logspectral) outperforms chroma in both accuracy and latency — inverting a common recommendation and giving me a concrete variable to test next.

The question stopped being "how do I build a score follower?" and became "does this one work on my audio, out of the box?"

III · The plan

A small, unambitious test.

The experiment I defined was deliberately minimal. The goal wasn't to ship anything — it was to answer one question: can Matchmaker, running on my specific test files, produce a monotonically-advancing alignment?

If yes, I'd pivot. If no, I'd fall back to a narrower scope (monophonic instruments only, a product decision rather than an engineering failure). Either answer was more valuable than another week of aubio tuning.

Step 01

Create an isolated conda environment.

Matchmaker requires Python 3.12; my existing turnthepage env is 3.11. New env — don't break what's working.

Step 02

Install Matchmaker and its audio dependencies.

Fluidsynth, PortAudio, glib via conda-forge; Matchmaker itself via pip.

Step 03

Prepare the test inputs.

Convert the piano recording from .m4a to .wav. Keep the existing .mxl score for the first attempt.

Step 04

Write a small, disposable test script.

No live mic. No UI. Just feed pre-recorded audio to Matchmaker and print the beat positions it produces.

Step 05

Look at the output honestly.

Monotonic numbers = success. Chaos = fail. Anything in between = a specific, named next problem.

IV · The instrument

Forty lines of Python.

The entire test lived in a single script, match_test_mm.py, sitting beside my existing aligner so nothing was overwritten. The Matchmaker API turned out to be about as simple as a library can be:

match_test_mm.py Python 
"""
Go/no-go experiment: can Matchmaker align
Idea No. 10 out of the box?
"""
from pathlib import Path
from queue   import Empty
from matchmaker import Matchmaker
 
SCRIPT_DIR       = Path(__file__).parent
SCORE_FILE       = SCRIPT_DIR / "idea-n10-gibran-alcocer-8bars.mxl"
PERFORMANCE_FILE = SCRIPT_DIR / "Idea10.wav"
 
mm = Matchmaker(
    score_file       = str(SCORE_FILE),
    performance_file = str(PERFORMANCE_FILE),
    input_type       = "audio",
    # method="arzt"          (default: OLTW variant)
    # feature_type="chroma"  (try "logspectral" next)
)
 
positions = []
try:
    for current_position in mm.run():
        positions.append(current_position)
        if len(positions) % 20 == 0:
            print(f"  frame {len(positions):4d}:  beat = {current_position:.2f}")
except Empty:
    print("\n(End of audio reached.)")

The whole thing is about forty lines, including comments and a summary block. The entire "intelligence" of the test is on two lines: mm = Matchmaker(...) and for current_position in mm.run(). Everything hard happens inside the library.

V · The install gauntlet

Setup pain, in the order it happened.

The setup was not clean. This is worth recording honestly — most of a day of "AI product design" is actually patience with broken tooling. Every one of these was solvable, but none of them was obvious.

Obstacle 1 — llvmlite refused to compile

pymatchmaker pulls in librosa, which pulls in numba, which pulls in llvmlite — a library that requires the LLVM compiler toolchain to build from source. My Mac doesn't ship one. Pip tried to build anyway and failed with a wall of red:

matchmaker — zsh
Building wheel for llvmlite (pyproject.toml) ... error
  FileNotFoundError: llvmlite needs CMake tools to build. It
  appears that the 'cmake' tool is either not installed or not
  found on the path.
ERROR: Failed building wheel for llvmlite
× Failed to build installable wheels for some pyproject.toml
  based projects
╰─> llvmlite

Fix: install numba (and therefore llvmlite) through conda-forge first. Conda ships a precompiled binary. Pip then saw "Requirement already satisfied" and moved past the broken build step.

Obstacle 2 — missing pandas dependency

The library ran, got far enough to import, then crashed on a missing import inside partitura (Matchmaker's MusicXML parser). Not Matchmaker's fault — a packaging oversight in a dependency. Another conda install, problem gone.

Obstacle 3 — MuseScore's "Save As" doesn't export MusicXML

Not a code problem, but a UI one. In MuseScore 4, Save As only offers native formats. To get .mxl out, you have to use File → Export. I caught this because the dialog I was staring at had only two options in its format dropdown, neither of which was what I wanted.

Lesson worth keeping

On this kind of project, "the hard part" is rarely the algorithm. It's the eight small integration friction points between tools that were never designed to work together. Budgeting time for them upfront is more realistic than pretending they don't exist.

VI · First run

Working, but pointing at the wrong thing.

The first successful run was both encouraging and confusing. The alignment was running — no crashes, numbers advancing forward — but the beat positions were huge. Frame 20 was already at beat 68. For a 12-second piano recording this made no sense.

  frame  220:  beat = 238.37

_queue.Empty

The progress bar gave me the clue: Matchmaker was reading my score as having 370 beats total. A short piano piece at a typical tempo has maybe 20–40 beats. The MusicXML file I was using contained the entire piece, not just the eight bars I'd recorded. The aligner was trying to match a 12-second performance against a three-minute score — and, remarkably, it was still finding the right region.

The _queue.Empty at the end wasn't a real failure. It was Matchmaker politely asking "is there more audio?" after my file ran out. I caught this exception in the next version and turned it into a clean end-of-stream message.

The alignment working on the wrong-sized score was a better signal than I realized. It meant the library was robust enough to partially succeed even when I fed it bad inputs.

VII · Trimming the score

Matching the instrument to the recording.

To get a fair test, the score and the performance needed to cover the same bars. I opened the .mxl in MuseScore, selected everything from bar 9 onwards, and deleted those measures entirely. What remained was exactly the eight bars I had played.

I exported it as idea-n10-gibran-alcocer-8bars.mxl, side-by-side with the original. Two files now: the full score for eventual use, and the trimmed score for today's test. Pointed the script at the new file and ran it again.

VIII · The result

Zero backward jumps.

The second run did what the first run promised. The output stream started near zero, climbed steadily, paused during held notes, and ended cleanly.

IX · What changed

Where I was, where I am.

Before

Hand-rolled aligner on monophonic pitch detection

  • 4–8 notes matched out of 8 bars played
  • Non-linear cursor: jumped backwards on fast passages
  • Hand-coded "forward-only ±15 note" window as a patch
  • No concept of tempo adaptation
  • Fixed 40 Hz pitch tolerance — poor at low registers
  • GPL-licensed dependency — unclear commercial path
After

Matchmaker (OLTW) on chroma features

  • Tracked ~90% of the piece on first working run
  • Zero backward jumps. Monotonicity by construction
  • OLTW does the job the hand-rolled window imitated
  • Tempo difference handled automatically (score 11.25s, perf 12.6s)
  • Relative feature matching — octave-robust by design
  • Apache 2.0 — commercially friendly
X · What it means for the project

A scope decision, not just a technical one.

The biggest shift from today isn't the library. It's the reframing of what I'm actually building.

The original plan had me spending months implementing real-time score following from scratch — which is a genuine open research problem, not a weekend project. With Matchmaker doing that heavy lifting, my work becomes something different and, honestly, more interesting for a product portfolio:

  • → UX for performing musicians. Look-ahead buffering, page-turn timing, teaching mode — the things that actually determine whether a pianist trusts the app on stage.
  • → Translation layer. Converting Matchmaker's beat-position output into measure numbers and page boundaries — the actual units a page turner needs.
  • → Robustness work. Environmental noise, tempo changes, rallentandi, what happens when the musician restarts mid-piece.
  • → Mobile delivery. The Flutter port, the app store path, the onboarding flow that makes this feel obvious to use.

None of those are research problems. They're product problems — and they're the problems I'm actually qualified to solve.

The most valuable thing today wasn't building anything. It was recognizing that something better already existed, and being willing to stop defending the work I'd already done.

Next experiments

On the roadmap for the next session: try feature_type="logspectral" per the ISMIR paper's finding and compare against chroma; test on a second piece (Una Mattina) to confirm the result generalizes; and build the beat-to-measure translation needed to rewire my existing Streamlit UI to Matchmaker's output.
